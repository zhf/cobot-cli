# Agent Notes

## Tooling
- This repo is Bun-first: use `bun install`, keep `bun.lock` committed, and do not recreate `package-lock.json`.
- The root package is still the CLI package; `packages/*` is only a reserved Bun workspace path for future packages.
- Useful checks are `bun run typecheck`, `bun run build`, `bun run start -- --help`, and `bun run start -- config get`. There are no test or lint scripts in `package.json`.
- `bun run dev` is `tsc --watch`; do not start it unless a watch process is explicitly requested.
- If testing the linked `cobot` command, verify `command -v cobot`; this workspace is expected to link through Bun at `~/.bun/bin/cobot`, not the old NVM/npm global bin.

## Runtime Shape
- Published CLI entry is `dist/bin/cobot.js`, built from `src/bin/cobot.ts`; the shebang is `node`, so the package still supports the Node runtime declared in `engines`.
- CLI flow is `src/bin/cobot.ts` -> `src/cli/startChat.ts` for interactive Ink UI, or `src/cli/runPrompt.ts` for `run`/legacy `--prompt`; both create `Agent` from `src/core/agent.ts`.
- Real top-level subcommands live in `src/bin/cobot.ts`: `run`, `init`, and `config`; in-app slash commands live separately under `src/commands/definitions/`.
- TypeScript uses `moduleResolution: NodeNext`; relative source imports intentionally include `.js` extensions even when importing `.ts` files.
- User config is stored at `~/.cobot/config.json` with priority `config file > COBOT_* env vars > OPENAI_* env vars`; config string values can contain `$VAR`/`${VAR}` placeholders expanded at read time.
- Config key `extraRequest` is a JSON object serialized as a string. It is merged into chat completion request bodies as defaults before code-controlled fields override it.
- The `explore` agent can optionally rerank evidence with `qwen3-rerank` (or any compatible DashScope rerank model) when `explore.rerank.model` and `explore.rerank.apiKey` are configured. Rerank is advisory: deterministic exact matches and per-role coverage minima are always preserved, and any rerank failure falls back to deterministic ordering. Configure via `cobot config set explore.rerank.model qwen3-rerank` plus `explore.rerank.apiKey` (or `DASHSCOPE_API_KEY`). Tunables: `explore.rerank.topN`, `explore.rerank.perRole`, `explore.rerank.timeoutMs`, `explore.rerank.instruct`, `explore.rerank.baseURL`.
- The `explore` agent supports adaptive one-round mode: after round 1, if role-bucket coverage, high-priority file count, declaration evidence, and worker confidence all meet gates, round 2 is skipped. This reduces wall-clock time by ~20-35s on high-confidence broad requests. Hard cases still run round 2. Gate thresholds are configurable via `explore.adaptive.minHighPriorityFiles` (default 4), `explore.adaptive.minDeclarationEvidence` (default 1), and `explore.adaptive.maxLowSignalRatio` (default 0.5). Env fallbacks: `COBOT_EXPLORE_ADAPTIVE_MIN_HIGH_PRIORITY_FILES`, `COBOT_EXPLORE_ADAPTIVE_MIN_DECLARATION_EVIDENCE`, `COBOT_EXPLORE_ADAPTIVE_MAX_LOW_SIGNAL_RATIO`.
- Multi-repo workspace scans (e.g. `~/Code` with many sibling project directories) use recent-first ordering when `explore.scan.recentFirst` is true (default) and the root has at least `explore.scan.multiRepoMinDirs` child directories (default 8). Each top-level repo gets a per-repo file budget (`explore.scan.perRepoMaxFiles`, default `max(1500, maxFiles/40)`) before moving to the next repo, so newer repos are not starved by alphabetical traversal hitting `explore.scan.maxFiles` (default 60000). Single-repo roots keep deterministic depth-first alphabetical traversal. Tunables: `explore.scan.maxFiles`, `explore.scan.recentFirst`, `explore.scan.multiRepoMinDirs`, `explore.scan.perRepoMaxFiles`. Env fallbacks: `COBOT_EXPLORE_SCAN_MAX_FILES`, `COBOT_EXPLORE_SCAN_RECENT_FIRST`, `COBOT_EXPLORE_SCAN_MULTI_REPO_MIN_DIRS`, `COBOT_EXPLORE_SCAN_PER_REPO_MAX_FILES`. Ndjson scanning progress may include `scanStrategy` and `reposScanned`.
- The `explore` agent validates cited paths after synthesis: any path in the final answer that does not exist in the local file list or evidence history is flagged as potentially hallucinated. This is the safety net for thinking-disabled synthesis.
- The `explore` agent supports split thinking configuration via `explore.thinking.worker` and `explore.thinking.synthesis` (each `default` or `disabled`). Setting `explore.thinking.worker` to `disabled` removes thinking overhead from worker calls (the largest measured speedup, ~25-35s on broad cases). Final synthesis defaults to `default` thinking; setting it to `disabled` is faster but relies on cited-path validation to catch hallucinations. Env fallbacks: `COBOT_EXPLORE_THINKING_WORKER`, `COBOT_EXPLORE_THINKING_SYNTHESIS`.
- `cobot run --output ndjson` emits structured newline-delimited JSON events to stdout for machine consumption. Each line is a JSON object with a `type` field: `progress` (explore pipeline stages: skills, scanning, reranking, workers, adaptive-check, synthesis, validation, fast-path), `tool` (tool start/end for non-explore agents), `result` (final content), or `error`. This allows caller processes to consume progress and results in a customized format. Default output mode is `text` (plain text to stdout, unchanged behavior).
- The `explore` agent auto-loads skills via harness-side activation: since the explore agent bypasses the tool loop, it cannot call the `skill` tool. Instead, `chatWithParallelExplore` matches installed skill descriptions against the user prompt using term-overlap heuristics (`findMatchingSkillsForPrompt` in `src/core/skills.ts`), then injects matching `SKILL.md` bodies into the conversation context via `formatConversationContext`. Matching skill names are emitted as a `{"type":"progress","phase":"skills","skillsLoaded":[...]}` ndjson event. Skill discovery follows symlinks in `~/.agents/skills/` and `~/.cobot/skills/` (user-level) plus `.agents/skills/` and `.cobot/skills/` (project-level).

## Adding Or Changing Tools
- Built-in tool schemas live in `src/tools/schemas/index.ts`; handlers live under `src/tools/`; dispatch is wired manually in `src/tools/registry.ts`.
- When adding a tool, update the schema list, import/register it in `TOOL_REGISTRY`, and add the argument mapping in `executeTool`.
- `docs/Plugins.md` is stale design/proposal material: plugin registration APIs mentioned there are not present in `src/`.

## Style And Files
- Respect `.editorconfig`: tabs by default, LF endings, final newline; YAML uses 2 spaces.
- `dist/` is generated and ignored. Rebuild with `bun run build` instead of editing built files.
- `.npmignore` excludes `bun.lock` from published files, but the repository should still track `bun.lock` for reproducible installs.
